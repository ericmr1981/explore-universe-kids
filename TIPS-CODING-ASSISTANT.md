# Tips — Coding Assistant（执行型 agent）

> 这是给“能力略弱于主会话”的执行 agent 的提示卡：少走弯路、少返工。

## 1) 先对齐：你要交付什么
- 只按 `TASK.md` 做事；不明确的地方先问 L2（主会话）。
- 任何生产改动（Nginx/VPS/systemd）不要碰，只给**命令清单**与**建议**。
- **标准流程提醒**：如果任务明确要求“标准流程/L3”，则 L3 必须用 **Claude Code（claude CLI）**；调用失败要显式报错并尝试修复，无法修复就通知 Eric（不要换别的执行器）。

## 2) 子路径（/xxx/）是第一优先级
- 默认假设部署在子路径，比如 `/space/`。
- 检查点：
  - HTML 引用的 CSS/JS 是否用相对路径？
  - JS 的 `fetch()` 路径在子路径下是否仍然正确？
  - 跳转链接是否带上 base path？

## 3) 登录继承（portal_auth）
- 只做“是否存在 cookie”的简单判断即可（别做复杂 JWT）。
- 未登录跳 `/login.html`。

## 4) 静态站点优先（能省掉 80% 的部署麻烦）
- 优先用 `public/` 下的纯静态资源：HTML + CSS + JS + JSON。
- 少引入构建工具，除非确实需要（否则 base path、构建产物路径会踩坑）。

## 5) apps.json 变更最容易翻车
- 任何 JSON 少一个逗号就会让 Portal 整体入口坏掉。
- 修改后务必：格式化 + 校验 JSON。

## 6) 验收要写“人类能按着点”的步骤
- 列 3~7 条：打开什么 URL、点哪里、看到什么。
- 报错时：给出复现步骤 + 控制台错误 + 推测原因。

## 7) 产出格式（固定）
- 变更文件列表（路径级）
- 关键 diff 摘要（不要贴全量）
- 通过/失败的门禁结果（命令 + 结果）
- smoke test 结果（URL + 截图可选）
